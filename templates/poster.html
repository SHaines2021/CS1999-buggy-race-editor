Task List Summary:

0-GET

I forked the repo on GitHub and then used git clone to make a local copy.

Once the repo was on my own machine I was able to commit changes to version control as I went along, and I pushed back up to my GitHub repo at the end of each task. I followed GitHub's instructions and set up a SSH key so I didn't need to enter username and password every time I pushed.

0-GET Get the source code

Logging into the buggy server, VSCode and then GitHub allowed for automatic population of the required files.

This task required starting from scratch- setting up a Github account, downloading and navigating VSCode for the first time.

0-RUN Get app running and view it in a browser

Opened VSCode workspace, successfully ran the app.py code and opened the server link.

0-CHANGE Make a change to a template and see it appear

Altered the heading of the Index page and witnessed the update in the live page.

Changed this to "Sarah's Buggy Editor".

1-TEMPLATE Add a new template to the app

Added an info page template to the app, this includes an iframe that links to the server so it does not go out of date if the racing server is updated.

Initially I looked into web-scraping this data with BeautifulSoup but it seemed like an excessive amount of effort for the desired effect. I then copied the original source code from the server in as a placeholder before replacing it with the iframe.

1-ADD Add more data to the form

Added flag_color to the form

This was done because it already existed in the database, ready for use.

1-VALID Add basic data validation

Added a simple statement to validate if the input for qty_wheels was a digit or not.

Also added in a flash message here to alert in the event of failure. This meant that the form page could be reloaded, rather than the update page displaying the fail and having to backtrack to fix it.

1-STYLE Style your editor just how you like it

Basic updates to the app style were made, particularly around the colour selections and font.

A minimalist, high contrast style was selected for this task.

2-EDIT Edit the record by loading its current values into the form

Added the current values into the form and rendered the template with these in place

This was done by reusing the SQL code for fetching the record from app.py and placing it in the GET method then rendering the template with this as the data

2-FORM Make the form better

Added in all the buggy attributes to the form, also added dropdown menus for all named fields to ensure valid entries and facilitate rapid form filling.

There are a lot of these! But as the races were not live at the time of writing, I wanted to ensure that the output of the app would result in a valid buggy that was ready for racing. The dropdown menus allow for pre-selected values, so the content of these was imported from the current buggy record. This works well but does mean that on first opening some items may be listed twice (ie. petrol, petrol, fusion, steam), but this does not alter its functionality.

2-COST Calculate and save the game cost of the buggy

Imported costs from the racing server JSON API and added if statements to handle 0 cost where items were not used.

The JSON import ensures that any cost updates will be immediately reflected in the buggy editor. It also allows for a dictionary-type look up of each element. Costs were also calculated by sub-category for ease of modification between race budgets.

2-RULES Add validation according to the game rules

Rules for the form data were added as if statements to ensure only valid entries were saved to buggies.

Each error results in a flash message about it's content (ie. "Oops! Please enter at least 0 attacks. (qty_attacks)") giving an outline of the error and it's location. This was later updated so that all the .isdigit functions were in the same loop. An attempt was made to put the results of these errors into a function, but it was not possible to get the template to render form the return of these.

3-ENV Switch between dev and production environments

Switched between the two environments successfully

Ran and tested both versions.

3-AUTOFILL Add auto-fill to the edit forms

Added buttons to the form that filled and submitted a buggy. Either a buggy with random values or one with default values.

This route was taken as the form was already filled with default values upon opening (from 2-edit), it would have been great to have this functionality through javascript as you fill out the form, however, this was not possible in the time. Buttons allow for the records to be populated and submitted but as it's server side rather than client side, the values aren't visible until the buggy is view in the summary page. The random values have set lists for colours and numbers so that the random.choice function could be used, but these are not exhaustive.

3-MULTI Allow different buggies to be created

Added the functionality to allow for multiple buggies

This was done in association with the course videos. This code meant that the auto-fill no longer functioned as it needed to pass a None record into the buggy creation template. This was fixed by updating the init_db.py file to include a complete default buggy (not just 4-wheels), then calling this into the form. This did, however, mean that where there was a data validation error it would reset this ID to 0, modifying this default. So code was added to delete the id before loading the template for data validation. This would have been better in a function, but as above, the template render did not work. Additionally, the buggy ID is hidden from the user and automatically increments. This means that no numbers are skipped.

3-DEL Allow buggies to be deleted

Added the functionality to delete individual buggies

This was done in association with the course videos. Due to the fact that the buggy IDs increment automatically, if buggies 1,2,3 existed and 2 was deleted, and a new buggy was created, this would increment to 1,3,4. If more time was available then functionality to review the IDs in use and back fill any missing ones would be desirable.

3-FLAG Display the pennant graphically

Flag colours and patterns displayed within the buggy summary page, within a cell of the table.

This was done through an if statement to determine the flag pattern style, then by selecting the correct table cell that featured the modified style of that pattern. The colours of the flag were pulled from the buggy record.

3-TESTS Write some tests

Manual walkthrough testing details are outlined in the testing.md document

Over the course of this task a large number of potential testing methods were attempted with the view to generating an automatic test set. This proved not to be possible within the remaining time. If more time were available then an automatic testing suite could be written in pytest or unittest generating a duplicate/test database and unit tests for each functional block of the application code and verification statements like AssertTrue(x) to determine a range of pass/fail criterions. For example, with the input of a dummy buggy record it would be possible to quickly test if the cost calculations, database and form were capable of functioning as anticipated. After this boarder/edge values could then be added to the trial to ensure greater stability. This may include a very large number of wheels, symbols or decimals.

Thanks for reading!
